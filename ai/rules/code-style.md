# Code Style

TypeScript strict mode.

Use the ESLint/Prettier configuration generated by `nest new` as the baseline; don't hand-roll a custom lint config without discussing it first.

Standard NestJS layering: controllers handle HTTP, services hold business logic, modules wire dependencies. Business logic does not live in controllers.

## Validate against the actual domain, not just the JS type

A validator that checks `typeof x === 'string'` or `typeof x === 'number'` is not the same as checking the field is actually valid. Two gaps `typeof` never catches:

- **Enum/union membership.** If a downstream schema constrains a field to a specific set of values (e.g. a Mongoose `enum:`), the validator must check membership in that same set — ideally importing the same constant, not re-typing the list — not just that the value is *a* string.
- **Finite numbers.** `typeof NaN === 'number'` and `typeof Infinity === 'number'` are both `true` in JavaScript. Anywhere a "number" field will be used in arithmetic, comparison, or persisted, validate with `Number.isFinite(x)`, not `typeof x === 'number'`.

See `docs/retrospectives/2026-07-backend-implementation-lessons.md` Pattern 1 for the concrete bugs this gap produced.
