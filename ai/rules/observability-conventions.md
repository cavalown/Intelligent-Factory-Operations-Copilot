# Observability Conventions

Structured logging, trace correlation, and the `ifoc.events.processed` metric are wired in globally (`backend/src/instrumentation.ts`, `nestjs-pino` in `app.module.ts`, `KafkaConsumerBase`) ‚Äî see `docs/design/observability.md` for how the subsystem works. New code gets these for free only by following the patterns below; stepping outside them loses the correlation/metric coverage silently, with nothing that errors to flag it.

- **Log via `Logger` (`@nestjs/common`), never `console.log`.** `console.log` bypasses the pino pipeline entirely ‚Äî no JSON, no `trace_id`, no shipping to Loki.
- **New Kafka consumers must extend `KafkaConsumerBase`**, not a hand-rolled `kafkajs` consumer ‚Äî extending it is what gives `ifoc.correlation_id`/`event_id`/`event_type` span attributes and the `ifoc.events.processed` counter automatically. `handleMessage` returns `Promise<boolean>`: `true` only when the event caused a real effect, `false` for every deliberate no-op (unrecognized type, idempotent duplicate, unknown machine, etc.). Returning `true` for a skip inflates the metric ‚Äî see `docs/retrospectives/2026-07-observability-review-lessons.md` Pattern 2 for what that looked like the first time it happened.
- **Code outside an HTTP request or a `KafkaConsumerBase` subclass (a cron job, a background worker, a new queue integration) has no active span by default.** Logs still come out as JSON, but won't carry a `trace_id` unless one is deliberately created ‚Äî don't assume trace correlation is universal.
- **A new custom metric needs an explicit OTel counter/histogram.** HTTP/Mongo/Kafka defaults are auto-instrumented; anything domain-specific (e.g. LLM call duration) isn't created for you.
- **Before adding any statement to an existing try/catch that classifies errors** (`isDataError` in `error-classification.util.ts`, used by every `KafkaConsumerBase` subclass), check what the new statement can throw against that classifier's enumerated list (`SyntaxError`, `ValidationError`, `CastError`). An unrecognized exception type gets rethrown into kafkajs's retry path instead of being swallowed ‚Äî the single most-repeated mistake in this subsystem's review history (`docs/retrospectives/2026-07-observability-review-lessons.md` Pattern 1).
