# Observability Conventions

Structured logging, trace correlation, and the `ifoc.events.processed` metric are wired in globally (`backend/src/instrumentation.ts`, `nestjs-pino` in `app.module.ts`, `KafkaConsumerBase`) — see `docs/design/observability.md` for how the subsystem works. New code gets these for free only by following the patterns below; stepping outside them loses the correlation/metric coverage silently, with nothing that errors to flag it.

- **Log via `Logger` (`@nestjs/common`), never `console.log`.** `console.log` bypasses the pino pipeline entirely — no JSON, no `trace_id`, no shipping to Loki.
- **New Kafka consumers must extend `KafkaConsumerBase`**, not a hand-rolled `kafkajs` consumer — extending it is what gives `ifoc.correlation_id`/`event_id`/`event_type` span attributes and the `ifoc.events.processed` counter automatically. `handleMessage` returns `Promise<boolean>`: `true` only when the event caused a real effect, `false` for every deliberate no-op (unrecognized type, idempotent duplicate, unknown machine, etc.). Returning `true` for a skip inflates the metric — see `docs/retrospectives/2026-07-observability-review-lessons.md` Pattern 2 for what that looked like the first time it happened.
  - **Exception — a consumer with no persistence of its own can't detect a redelivered duplicate.** `rules-service-group` (`add-rule-engine/design.md`) is stateless by design (`module-boundaries.md`) and has no `lastEventId`/duplicate-key check to catch a redelivery, unlike every other consumer here. It still returns `false` for the no-ops it *can* detect (unknown machine, non-finite temperature), but a crash between publishing and offset-commit will re-increment its counter for an event it already counted. This is a deliberate, scoped trade-off, not a bug to fix by adding new state — see `add-rule-engine/design.md`'s Risks section.
- **Code outside an HTTP request or a `KafkaConsumerBase` subclass (a cron job, a background worker, a new queue integration) has no active span by default.** Logs still come out as JSON, but won't carry a `trace_id` unless one is deliberately created — don't assume trace correlation is universal.
- **A new custom metric needs an explicit OTel counter/histogram.** HTTP/Mongo/Kafka defaults are auto-instrumented; anything domain-specific (e.g. LLM call duration) isn't created for you.
- **Before adding any statement to an existing try/catch that classifies errors** (`isDataError` in `error-classification.util.ts`, used by every `KafkaConsumerBase` subclass), check what the new statement can throw against that classifier's enumerated list (`SyntaxError`, `ValidationError`, `CastError`). An unrecognized exception type gets rethrown into kafkajs's retry path instead of being swallowed — the single most-repeated mistake in this subsystem's review history (`docs/retrospectives/2026-07-observability-review-lessons.md` Pattern 1).
