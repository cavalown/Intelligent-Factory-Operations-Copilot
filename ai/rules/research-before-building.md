# Research Before Building

Before designing custom logic for a cross-cutting infrastructure concern — retry, backoff, timeout, connection handling, rate limiting, error recovery — check whether a library already in use (Kafka client, MongoDB driver, NestJS itself) already provides it. Read enough of that library's actual behavior (source or docs, not just a config option's name) to know *how* it works, specifically how it interacts with error propagation and control flow. Only build something bespoke after confirming the existing mechanism doesn't fit, and say so explicitly in the design doc.

When a design doc claims two pieces of code "don't interact" or "are independent" and both are part of the same runtime call stack (even across a library boundary), that claim must be verified by tracing actual control flow — not asserted from what the new code does or doesn't import. If it can't be quickly verified, mark it as an open question rather than a stated fact.

**Why:** `docs/retrospectives/2026-07-backend-implementation-lessons.md` Patterns 3-4 — a hand-rolled Kafka consumer retry mechanism was designed, implemented, and shipped based on a design-doc claim ("no interaction with kafkajs's own retry config") that was never actually checked against the library's source. It was false: the new code made the Kafka client's own built-in, jittered, already-tested retry mechanism permanently unreachable, and introduced a session-timeout/rebalance risk the built-in mechanism doesn't have. The fix required a full architectural reversal after the fact. The same session correctly grepped the *current codebase* for reuse opportunities (e.g. consolidating a duplicated `isDuplicateKeyError` check) — the reflex just didn't extend to checking the underlying *dependency's* behavior before building something new on top of it.
