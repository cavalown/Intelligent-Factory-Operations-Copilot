# 後端實作教訓（2026-07）

涵蓋從 walking skeleton 建起 IFOC 後端、歷經 3 輪 `/code-review` 與一次完整架構翻案（Kafka consumer 錯誤處理：一個手工打造的重試機制被實作、驗證可運作，之後因 code review 發現它與 Kafka client 函式庫自身的重試機制根本性衝突而被撤銷）的那個工作階段。本文件以**根因模式**組織，而非以 bug 組織，因為 3 輪審查中約 20 個個別發現大多可以歸納為 4-5 個重複出現的錯誤。

---

## 模式 1：驗證是照著規格的範例寫的，而不是照著欄位完整的約束空間寫的

**由此產生的 bug：**
- `STATUS_CHANGED` 的 `currentStatus` 只驗證「是字串」，從未對照實際的 5 值 `MachineStatus` enum 檢查，所以像 `"BOGUS"` 這種 enum 之外的值通過了 HTTP 驗證、抵達 Kafka consumer，然後在儲存時拋出 Mongoose `ValidationError`。
- `schemaVersion` 用 `typeof x !== 'number'` 驗證，這無法拒絕 `NaN` 或 `Infinity` — 在 JavaScript 裡兩者的 `typeof` 都是 `number`。像 `1e400`（合法的 JSON 語法，經浮點溢位解析為 `Infinity`）這種值就溜過去了。
- `TEMPERATURE_REPORTED` 的 `temperature` 有同樣的「只查 `typeof`」缺口，而且修復最初只套用到讀取它的兩個 consumer 之一（為什麼局部修復一再發生，見下方模式 3）。

**為什麼會發生：** 每個驗證器都是看著 `docs/design/event-schema.md` 的「必要 payload 欄位」表、問「這個欄位存在嗎？JS 型別大致對嗎？」寫出來的 — 這能正確抓到缺欄位，但從來沒問「這個欄位*實際*的值域是什麼？」`currentStatus` 的值域不是「一個字串」，而是「5 個特定字串之一」，定義在*另一個*檔案（`machine-schema.md` / `machine.schema.ts` 的 `MACHINE_STATUSES`）。在這個 codebase 裡，「number」欄位的值域不是「任何 `typeof` 說是 number 的東西」，而是「有限數值」— 這個區別 JavaScript 不會幫你做，只有刻意把 `NaN`/`Infinity` 當成輸入來思考（而不只是想錯誤型別的輸入）才會浮現。

**如何避免：** 為欄位寫驗證器時，不要停在「schema 文件說這個欄位必填、型別大概是什麼」。要問：這個欄位的*值*在下游哪裡被消費？*那段*程式碼假設了什麼？如果下游的 Mongoose schema 有 `enum:` 約束，驗證器就該檢查是否屬於同一個 enum（最好 import 同一個常數，而不是重打一遍清單）。如果下游計算會對「number」做算術或比較，就用 `Number.isFinite()` 驗證，而不是 `typeof === 'number'`。

## 模式 2：修掉一個被回報的邊界情況，卻沒有重新檢查函式完整的輸入空間

**Bug 本身：** `events.service.ts` 的分頁 `limit` 參數有個已知 bug：`Number(query.limit) || DEFAULT_LIMIT` 把 `limit=0` 當成 falsy，無聲地回傳預設頁面大小而不是 0 筆的頁面。修復 — `query.limit !== undefined ? Number(query.limit) : DEFAULT_LIMIT` — 正確修掉了 `0` 的情況，但也**無聲地重新引入了另一個 bug**：非數字的 `limit`（例如 `?limit=abc`）現在會產生 `Number('abc')` = `NaN`，不受任何防護地傳進 MongoDB 查詢。*原本*有 bug 的程式碼恰好把這個情況處理對了，因為 `NaN` 也是 falsy — 這個修復拿一個邊界情況換了另一個，直到下一輪 `/code-review` 才有人發現。

**為什麼會發生：** 修復的範圍被界定為「讓 `limit=0` 可用」，用 `limit=0` 測試驗證後就算完成。它從未針對原本那一行單行程式碼恰好也處理到的*其他*邊界情況（非數字字串、`undefined`、負數）重新測試 — 因為那些不是被回報的 bug，所以不在這次修復自身驗證的範圍內。像 `Number(x) || DEFAULT` 這種精簡的單一運算式，悄悄地同時編碼了好幾個不相關邊界情況的處理（falsy 的零、`NaN`、`undefined` 全部塌縮到同一個分支）；把它換成更明確的寫法，需要刻意重新推導所有這些情況的完整行為表，而不只是正在修的那一個。

**如何避免：** 當修復改變了函式處理某個邊界情況的方式時，明確列舉*舊*程式碼隱含處理到的（哪怕是意外處理到的）其他邊界情況，並確認*新*程式碼仍然全部處理 — 不要只驗證 bug 回報裡的那一個情況。原始程式碼是單一稠密運算式（`a || b`、`a ?? b`、用 truthy 檢查代替多個條件）時尤其如此 — 稠密往往正是多種情況能不被注意地塌縮進一行的原因。

## 模式 3：設計自訂的基礎設施邏輯前，沒有先確認已在使用的函式庫是否已經提供

**Bug 本身（最大的那個）：** 為了修復 Kafka consumer 在暫時性失敗（例如 MongoDB 短暫斷線）時無聲丟棄事件的問題，一個有上限的重試加退避（backoff）工具（`withRetry`，3 次嘗試、指數延遲）被設計、實作、手動驗證可運作並交付 — 是使用者在一次設計討論後從 3 個成文選項中選出來的。第二輪 `/code-review` 隨後發現：
1. **`kafkajs`（已是專案依賴）針對這一類失敗有自己內建、經過測試、帶抖動（jitter）的重試機制**（`Runner` 層對 `eachMessage` 錯誤的重試，可透過 `kafka.consumer({ retry: {...} })` 設定，預設 5 次帶抖動重試）。手工重試不只是重複了它 — 還**讓它永遠不可達**，因為包住 `eachMessage` 的外層 `try/catch` 吞掉了每一個錯誤，錯誤根本無法傳播到 kafkajs 自身的重試層。
2. 手工重試沒有抖動（固定退避），造成同步重試風暴的風險，而 kafkajs 自身的機制已經避免了這點。
3. 在單一 `eachMessage` 呼叫內重試、且重試延遲期間不呼叫 `heartbeat()`，在真正（而非短暫）的停機時有超過 consumer session timeout 的風險，會觸發 **group rebalance** — 一個由修復本身引入的新失敗模式，而且恰好出現在修復想幫忙的情境裡。

該決策被翻案：手工重試被刪除，改為小得多的做法：「分類錯誤，只有在重試絕對無濟於事時才自己吞掉；否則讓它傳播給 kafkajs 自身的重試」— 完全不需要自訂重試迴圈。

**為什麼會發生：** 問題被框成「我們要怎麼在 Kafka 訊息 handler 裡加重試邏輯」，這是一個*建造*某物的請求。沒有人在開始設計前先問「我們已在使用的 Kafka client 函式庫是不是已經解決了這個問題」。這是最開頭第一步的範圍界定失敗：最初的設計文件（`design.md`）提出 3 個選項 — 接受、分類、或重試 — 甚至（在選項 C 自己的說明裡）明確宣稱「與 kafkajs 自身的 consumer 層重試設定沒有互動」，把這當成手工方案的*優點*（更簡單、自成一體），而不是去確認它是否為真。這句話在寫成設計事實之前，沒有對照函式庫原始碼檢查過 — 見模式 4。

值得注意：*同一個工作階段*在較小的範圍正確運用了「檢查既有可重用的東西」— `isDuplicateKeyError` 在兩個檔案重複被抓到並整併為共用工具，理由正是它重複了*同層的應用程式碼*。「grep *目前 codebase* 找既有 helper」的反射動作是存在且有效的。缺的是「在設計一整個新機制前，先查*底層依賴*文件／原始碼層級的行為」的反射 — 重用檢查只套用在「我們是不是已經寫過這個」的層級，沒有套用在「我們依賴的函式庫是不是已經解決這個」的層級。

**如何避免：** 在為橫切的基礎設施關注點（重試、退避、逾時、連線池、速率限制、斷路器）設計自訂邏輯之前，明確地研究已在使用的函式庫（這裡是 `kafkajs`）是否有第一級支援，並閱讀足夠多的函式庫原始碼／文件以了解它*如何*運作（不只是知道*有*一個名字看起來對的設定選項）— 特別是它與錯誤傳播的互動方式，因為「這個函式庫會幫我重試嗎」是一個關於控制流的問題，不只是設定問題。只有在確認既有機制不合用之後才設計客製方案，並在設計文件中明說（「kafkajs 自身的重試在此不適用，因為 X」）。

## 模式 4：設計文件中關於函式庫行為的技術宣稱是被斷言的，不是被驗證的

**Bug 本身：** `design.md` 選項 C 的段落寫道，手工重試「與 kafkajs 自身的 consumer 層重試設定沒有互動 — 它完全在單一 `eachMessage` 呼叫內運作，在 kafkajs 自身的重試/心跳機制介入之前」。這正是後來被證明錯誤的那句話（見模式 3）— 而它被當成既定事實／取捨寫下並呈給使用者、影響了對方的決策，卻沒有實際追過 kafkajs 的原始碼確認。

**為什麼會發生：** 這個宣稱從表層推理*聽起來*是對的：新程式碼沒有從 `kafkajs` 的 retry 模組 import 任何東西，直覺上似乎與它無關。但兩段程式碼是否「互動」是一個關於執行期控制流的問題（某處拋出的錯誤是否會在某個要緊的地方被接住），不是關於 import 語句的問題。從「這段程式碼看得見地碰了什麼」而非「這段程式碼*沒有拋出*這件事阻止了別處的什麼發生」去推理，產出了一個聽起來合理但錯誤的結論。

**如何避免：** 任何形如「X 不影響 Y」或「X 與 Y 彼此獨立」的設計文件宣稱，只要 X 與 Y 同屬同一條執行期呼叫堆疊（即使是間接的、即使跨越函式庫邊界），都需要靠實際追蹤控制流來驗證 — 不能從新程式碼 import 或沒 import 什麼來斷言。如果宣稱無法快速驗證，就明說（「假設、未驗證 — 見 Open Questions」），而不是當成已定案的事實陳述。這與 `/code-review` skill 自身強制的紀律相同（CONFIRMED 需要引用實際的程式行／原始碼，不能用推論）— 這個紀律在*撰寫*設計文件時也該適用，不只是在對照設計審查程式碼時。

## 模式 5：一個潛伏整個工作階段的 bug，直到一種新型檢查（真正的單元測試）終於觸發它才被發現

**Bug 本身：** `Machine.status`、`Alert.severity` 與 `Alert.status` 是以字串字面值聯集（例如 `MachineStatus`）標型的 Mongoose `@Prop()` 欄位。當型別定義在*另一個檔案*、並透過較嚴格的 `import type` / `isolatedModules` 語意被使用時，`@nestjs/mongoose` 的裝飾器需要明確的 `type: String` 才能在 TypeScript 的 `emitDecoratorMetadata` 下正確解析 — 少了它，裝飾器會在類別實際載入的那一刻拋出「Cannot determine a type」。這個問題從最初的 walking-skeleton commit 就存在，卻從未浮現，因為完整的 NestJS 應用程式（跨整個編譯後程式解析型別，而非逐檔）從未碰到這個失敗模式，也沒有任何測試曾單獨 import 這些 schema 類別 — 直到第一個真正的單元測試（`sensor-failure-contract.spec.ts`，為了不相關的原因而寫 — 一個商業邏輯契約測試）在 `ts-jest` 下傳遞性地載入它們。

**為什麼會發生：** 運行中的應用程式（Docker 容器、`nest build`、`nest start`）從來沒有觸發過這條程式路徑 — 全程式編譯恰好能正確解析型別，所以 `npm run build` 與實際運行的伺服器一直都好好的。這個 bug 對當時使用的每一種驗證方法（對真實運行系統的手動 HTTP 測試、`docker logs`、`tsc --noEmit`）都是隱形的，因為它們都不會像 `ts-jest` 的逐檔轉譯那樣單獨載入單一檔案。

**如何避免：** 這其實無法靠事先寫更好的程式碼避免 — 這一類 bug（依編譯模式而異的 metadata 解析）就是需要能抓到它的那*種*檢查（單獨 import 該檔案的真正單元測試）先存在才可能被發現。可行動的教訓比較窄：當專案運行了一段時間而完全沒有真正的單元測試時（本專案正是如此，見 `ai/rules/testing.md` 的「策略未定案」註記），要預期加入*第一個*單元測試很可能純粹因為它是新型檢查而翻出至少一個不相關的潛伏問題 — 引入測試基礎設施時為此預留時間，發生時不必驚訝或擔心；這是新檢查在盡職的跡象，不是什麼東西退化的跡象。

## 後設教訓：真正抓到這些 bug 的是什麼，沒抓到的又是什麼

上述 5 個模式沒有一個是被以下方法抓到的：
- 對真實運行系統的手動 HTTP/Kafka 測試（全程大量且正確地使用 — 每次都確認了每個*功能*如預期運作）
- `tsc --noEmit` / `npm run build` / `npm run lint`（每一輪都乾淨）
- 作者寫完程式碼後立即的自我重讀

全部 5 個都是由一次獨立、系統性的 `/code-review` 抓到的 — 具體來說是那些不只問「這有沒有做到它該做的」（手動測試已回答）的視角，而是問「這*移除或改變*了什麼別的東西所依賴的行為」（removed-behavior auditor）、「這段程式碼與其依賴的實際邊界上會發生什麼」（cross-file tracer；模式 3/4 則是追進 `kafkajs` 函式庫自身的原始碼），以及「這是正確深度的修復，還是 OK 繃」（altitude）。

**推論：** 對基礎設施層級或橫切性的變更而言，對運行系統的手動驗證是必要但不充分的 — 它證明了 happy path 與你想到要測的特定情境，但結構上不可能抓到「這與別的東西的行為衝突」或「這沒涵蓋你沒想到要測的輸入空間」，因為你只會測你已經懷疑可能有錯的地方。對任何觸及共用基礎設施的變更（Kafka consumer 基底類別、跨事件類型共用的驗證、任何可能已有現成函式庫解決的東西），把一次 `/code-review` — 或至少刻意研究底層函式庫的行為 — 當作*完成*該變更的一部分，而不是「做完」之後可選的額外步驟。
