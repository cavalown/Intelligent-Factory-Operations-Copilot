# Dashboard Metrics Review Lessons (2026-07)

Covers the `/code-review` pass on the `dashboard-operational-metrics` change (status-transition projection, rolling-24h utilization, dashboard aggregates). The review confirmed 10 findings; this document records the 4 root-cause patterns behind the ones that matter, because each is a mistake class that would recur in a different shape if only the individual bugs were patched. Fixes shipped in the same change (design.md D6–D10).

---

## Pattern 1: A secondary write was placed on the primary write's critical path, and its failure modes were never played against the pipeline's error-classification policy

**The bug (the big one):** The projection consumer gained a second write — recording a status transition into `machine_status_transitions` — inserted *before* `machine.save()`, with a catch that swallowed only duplicate-key errors. The code comment justified the ordering by reasoning about **transient** failure and redelivery ("if the save fails and the message is redelivered, the duplicate-key swallow keeps this idempotent"). What it never considered: a **data** error (e.g. a mongoose `ValidationError` from a malformed `occurredAt` failing the schema's `required` check) rethrown from the transition write is classified by the consumer base's poison-pill mechanism as unprocessable — which logs, commits the Kafka offset, and moves on. Result: `machine.save()` never runs, and that event's status/health/production effect is **permanently lost** with no redelivery. A projection that is supposed to be rebuildable-from-events silently diverged from the event log.

**Why this happened:** Two mechanisms, each individually correct and individually well-reasoned, interacted destructively. The poison-pill skip (from a previous change, itself the product of a careful design reversal) is right: retrying an unprocessable message forever stalls the consumer. The write-ordering reasoning (transition-before-save, dup-key-swallow) is right *for the failure class it considered*. The failure was in never asking the composition question: **"for each class of error my new code can throw, what does the surrounding infrastructure do with it?"** The comment on the hunk proves the author reasoned about failure — but only enumerated the failure class that motivated the ordering choice, not the classes the error-classification layer distinguishes. The classifier's categories (data vs. transient) existed in the same codebase, documented, one directory away.

There is also a rank-ordering error underneath: transitions are a *derived, rebuildable, secondary* projection. Nothing secondary should ever be able to abort the primary projection update — that priority relationship was implicit in the design ("rebuildable projection") but not enforced in code. When a subordinate write can throw into a shared error path, its subordination is fiction.

**How to avoid it:** When adding a write (or any throwing operation) inside a pipeline that has an error-handling *policy* (poison-pill classification, retry tiers, dead-letter routing), enumerate your new code's failure classes against each branch of that policy — literally read the classifier and ask what it does with a ValidationError, a CastError, a network error thrown *from your line*. And when a design labels data as "secondary" or "rebuildable," make the code structurally agree: best-effort writes must catch-and-log everything, so their failure cannot change the primary path's control flow. If the two mechanisms live in different files, that is a reason for *more* scrutiny, not less — this is the same lesson as the previous retrospective's Pattern 4 (interaction is about runtime control flow, not proximity or imports), recurring one level higher: not "library vs. my code" this time, but "my previous change vs. my new change."

## Pattern 2: A weakly-validated field accumulated downstream consumers, each stacking new assumptions on a guarantee nobody ever established

**The bugs this produced (3 of the 10 findings):**
- Utilization's interval walk did `Date.parse(transition.at)` with no NaN guard — one unparseable timestamp poisoned every bucket (`Math.max(0, NaN)` is `NaN`) and the response rendered as "NaNh NaNm".
- The walk (and the window range query) assumed lexicographic string order equals chronological order — true only for canonical `...Z`-form ISO strings, which nothing enforced; an offset-form timestamp (`+00:00`) silently dropped elapsed time via the clamp.
- The 24h production sum used `$gte` with no upper bound, counting future-dated events that the duration computation (clamped to `now`) excluded — two numbers on the same dashboard row describing "the same window" could disagree.

**Why this happened:** `occurredAt`/`producedAt` were validated at ingestion as "not null" — nothing more — since the walking skeleton. That was *survivable* while consumers only stored and displayed the strings. This change made the field **load-bearing in a new way**: sorting by it, subtracting it, range-querying it. Each of those operations imports an assumption (parseable; canonically formatted; not in the future) that the ingestion boundary never promised. The compounding dynamic is the point: assumption debt on a weakly-validated field grows with every new consumer, and the consumer's author sees only their own code — the missing guarantee lives at a boundary written months (or sessions) earlier and looks like someone else's settled decision.

Notably, the *previous* retrospective's Pattern 1 ("validate against the field's actual domain, not just the JS type") had already been learned and turned into a rule (`ai/rules/code-style.md`) — and this session correctly applied it to enums and finite numbers elsewhere. But the internalized rule was scoped to enum membership and `Number.isFinite`; nobody generalized it to **string formats**. A timestamp's domain isn't "a string" any more than `currentStatus`'s domain was — it's "a canonical ISO-8601 UTC instant," and this codebase additionally relies on the *lexicographic-order property* of exactly that canonical form.

**How to avoid it:** When new code starts *computing* on an existing field — sorting, arithmetic, range comparison, grouping — stop and audit what actually guarantees that field's format, at the boundary where it enters the system. "It's always looked like an ISO string in the data" is an observation, not a guarantee. If the guarantee doesn't exist, add it at ingestion (this round: canonical-format validation with the existing `INVALID_EVENT_ENVELOPE` code) *and* keep cheap defense-in-depth at the consumer (parse guard, numeric re-sort) because Kafka has more producers than the validated HTTP path. And when two computations claim to describe the same window/set/period, make them share the boundary definition — both bounds, same source — rather than each independently encoding half of it.

## Pattern 3: Enumerated registries in docs don't update themselves when the code grows a new member

**The bugs this produced (2 findings):** the new `machine_status_transitions` collection was absent from `ai/context/mongodb-collections.md`, and the new `dashboard/` module was absent from both `architecture.md` §14.1's module tree and `ai/rules/module-boundaries.md`'s enumerated list — the rule doc effectively read as a closed whitelist that the shipped code violated.

**Why this happened:** The change *did* practice doc-sync — api.md gained full endpoint sections, the contract summary was updated — but only for the docs the change was *consciously writing against*. Registries are different: they index artifacts by **kind** (all collections, all modules, all error codes), so the trigger isn't "I touched this doc's subject," it's "I created a new instance of a kind some doc enumerates." That trigger fires at creation time in the code, far from any doc, and nothing in the workflow pointed back. The dashboard-module case had an extra wrinkle: the module was a *deliberate, well-reasoned* design amendment (a composition layer per architecture.md §7.7, resolving a module cycle) — recorded in the change's own design.md, but not propagated to the two standing docs that enumerate the module set, so the repo's rules and the repo's code disagreed the moment the change shipped.

**How to avoid it:** When a change creates a new instance of an enumerable kind — a module, a Mongo collection, an event type, an error code, an env var — grep the docs for where that kind's existing members are listed, and extend the list in the same change. The registries in this repo, as of this writing: `architecture.md` §14.1 (modules, mirrored by `module-boundaries.md`), `ai/context/mongodb-collections.md` (collections), `api.md` §6 (error codes), `event-schema.md` §4 (event types), `docker-compose.md` §5 (env vars). A deliberate design deviation recorded only in the change's design.md is half-recorded: the standing rule docs are what the *next* session reads first.

## Pattern 4: Derived numbers were presented with more authority than their inputs could support

**The bug (finding 9, and honorable mentions):** the bootstrap approximation — a machine with no transition history gets its whole 24h window attributed to its current status — produced responses byte-identical to fully-derived data. A machine running for 40 minutes could report "Operating 24h" with nothing marking it as estimated. Smaller versions of the same pattern: `formatDuration` rendered an exact 0 as "<1m" (implying a small nonzero amount), and an unparseable timestamp rendered as "NaNd ago" rather than an honest "—".

**Why this happened:** The approximation itself was a *documented, deliberate* decision (design D2, noted in api.md prose) — the mistake wasn't making it, but confining its existence to prose humans read while the payload machines read carried no trace of it. Data quality was treated as a documentation concern instead of a data concern. The implicit assumption: consumers of the API will have read §4.12's caveat. The dashboard — built in the same change — didn't, because there was nothing in the response to react to.

**How to avoid it:** When a computation has a degraded mode (approximation, fallback, partial coverage, stale cache), the *response* should say so — a boolean or a coverage timestamp costs one field and lets every consumer render honestly (`≈` in this fix). The test: if the UI wanted to visually distinguish estimated from real, could it, without parsing documentation? If no, the degraded mode is silent, and silent degraded modes eventually get read as facts.
