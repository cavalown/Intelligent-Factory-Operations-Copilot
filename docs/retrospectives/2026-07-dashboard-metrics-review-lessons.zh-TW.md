# 儀表板指標審查教訓（2026-07）

涵蓋 `dashboard-operational-metrics` 變更（狀態轉移投影、滾動 24 小時稼働率、儀表板聚合）的 `/code-review`。該次審查確認了 10 個發現；本文件記錄其中要緊者背後的 4 個根因模式，因為每一個都是一類錯誤 — 如果只修補個別 bug，它會換個形狀再次出現。修復已在同一變更中交付（design.md D6–D10）。

---

## 模式 1：次要寫入被放上主要寫入的關鍵路徑，且它的失敗模式從未對照管線的錯誤分類政策推演過

**Bug 本身（最大的那個）：** 投影 consumer 多了第二個寫入 — 把狀態轉移記錄進 `machine_status_transitions` — 插在 `machine.save()` *之前*，catch 只吞 duplicate-key 錯誤。程式碼註解用**暫時性**失敗與重送來論證這個順序（「如果 save 失敗、訊息被重送，duplicate-key 吞掉讓這保持冪等」）。它從未考慮的是：從轉移寫入重新拋出的**資料**錯誤（例如格式錯誤的 `occurredAt` 沒通過 schema 的 `required` 檢查而產生的 mongoose `ValidationError`），會被 consumer 基底的 poison-pill 機制分類為不可處理 — 於是記 log、提交 Kafka offset、繼續前進。結果：`machine.save()` 永遠不會執行，該事件的狀態／健康分數／生產效果**永久遺失**，不會重送。一個本該可從事件重建的投影，無聲地偏離了事件日誌。

**為什麼會發生：** 兩個機制，各自正確、各自論證充分，卻以破壞性的方式交互作用。poison-pill 跳過（來自前一個變更，本身還是一次審慎設計翻案的產物）是對的：對不可處理的訊息永遠重試會卡死 consumer。寫入順序的論證（先轉移後 save、吞 dup-key）*對它所考慮的失敗類別*也是對的。失敗在於從來沒問那個組合問題：**「對我新程式碼可能拋出的每一類錯誤，周圍的基礎設施會怎麼處置它？」** 那段 hunk 上的註解證明作者確實思考過失敗 — 但只列舉了促成順序選擇的那一類失敗，沒有列舉錯誤分類層所區分的各類別。分類器的類別（資料 vs. 暫時性）就存在同一個 codebase 裡，有文件、就在一個目錄之外。

底下還有一個優先序錯誤：轉移是*衍生的、可重建的、次要的*投影。任何次要的東西都不該有能力中止主投影更新 — 這個優先關係在設計中是隱含的（「可重建的投影」），但沒有在程式碼中強制。當一個從屬寫入可以往共用錯誤路徑拋出時，它的從屬性只是虛構。

**如何避免：** 在一條擁有錯誤處理*政策*（poison-pill 分類、重試層級、死信路由）的管線內新增寫入（或任何會拋出的操作）時，把你新程式碼的失敗類別逐一對照該政策的每個分支 — 真的去讀分類器，問它對*從你那一行*拋出的 ValidationError、CastError、網路錯誤各會怎麼做。而當設計把某份資料標為「次要」或「可重建」時，讓程式碼在結構上同意這件事：best-effort 寫入必須 catch 並 log 一切，使它的失敗不可能改變主路徑的控制流。如果兩個機制在不同檔案裡，那是*更*要仔細檢查的理由，不是更不用 — 這與上一份回顧的模式 4 是同一課（互動是關於執行期控制流，不是關於鄰近性或 import），只是升了一級再現：這次不是「函式庫 vs. 我的程式碼」，而是「我的前一個變更 vs. 我的新變更」。

## 模式 2：一個弱驗證的欄位累積了下游消費者，每一個都往一個從未被建立的保證上疊新假設

**由此產生的 bug（10 個發現中的 3 個）：**
- 稼働率的區間走訪對 `Date.parse(transition.at)` 沒有 NaN 防護 — 一個無法解析的時間戳毒化了每個 bucket（`Math.max(0, NaN)` 是 `NaN`），回應渲染成「NaNh NaNm」。
- 走訪（以及視窗範圍查詢）假設字典序字串順序等於時間順序 — 只有標準 `...Z` 形式的 ISO 字串才成立，而沒有任何東西強制這點；帶偏移量形式的時間戳（`+00:00`）會透過 clamp 無聲地丟失經過的時間。
- 24 小時生產總和用了沒有上界的 `$gte`，把未來日期的事件也算進去，而時長計算（clamp 到 `now`）排除了它們 — 同一儀表板列上描述「同一個視窗」的兩個數字可能不一致。

**為什麼會發生：** `occurredAt`/`producedAt` 從 walking skeleton 起在攝取（ingestion）時只驗證「非 null」— 僅此而已。在消費者只儲存與顯示這些字串時，這還*撐得住*。這次變更讓這個欄位以**新的方式承重**：拿它排序、相減、做範圍查詢。每一種操作都引入一個攝取邊界從未承諾過的假設（可解析；標準格式；不在未來）。複合的動態正是重點：弱驗證欄位上的假設債隨每個新消費者成長，而消費者的作者只看得到自己的程式碼 — 缺失的保證位於幾個月（或幾個工作階段）前寫下的邊界，看起來像是別人已定案的決定。

值得注意的是，*上一份*回顧的模式 1（「對照欄位實際的值域驗證，不只是 JS 型別」）已經被學起來並轉成規則（`ai/rules/code-style.md`）— 而且這個工作階段在別處正確地把它套用到 enum 與有限數值上。但內化的規則被限縮在 enum 成員資格與 `Number.isFinite`；沒有人把它泛化到**字串格式**。時間戳的值域不是「一個字串」，正如 `currentStatus` 的值域不是 — 它是「標準 ISO-8601 UTC 時刻」，而且這個 codebase 還額外依賴正是那個標準形式的*字典序性質*。

**如何避免：** 當新程式碼開始對既有欄位做*計算* — 排序、算術、範圍比較、分組 — 停下來，稽核在該欄位進入系統的邊界上到底是什麼保證了它的格式。「資料裡它看起來一直都是 ISO 字串」是觀察，不是保證。如果保證不存在，就在攝取層加上（這一輪：用既有的 `INVALID_EVENT_ENVELOPE` 代碼做標準格式驗證），*同時*在消費端保留便宜的縱深防禦（解析防護、數值重排序），因為 Kafka 的 producer 比走過驗證的 HTTP 路徑多。而當兩個計算宣稱描述同一個視窗／集合／期間時，讓它們共用邊界定義 — 兩個界線、同一個來源 — 而不是各自獨立編碼一半。

## 模式 3：文件裡的列舉式註冊表不會在程式碼長出新成員時自己更新

**由此產生的 bug（2 個發現）：** 新的 `machine_status_transitions` collection 沒有出現在 `ai/context/mongodb-collections.md`，新的 `dashboard/` 模組也同時缺席於 `architecture.md` §14.1 的模組樹與 `ai/rules/module-boundaries.md` 的列舉清單 — 那份規則文件實際上讀起來像一份封閉白名單，而已交付的程式碼違反了它。

**為什麼會發生：** 這個變更*確實*做了文件同步 — api.md 新增了完整的端點章節、契約摘要也更新了 — 但只針對變更*有意識地對照撰寫*的那些文件。註冊表不一樣：它們按**種類**索引產出物（所有 collection、所有模組、所有錯誤代碼），所以觸發條件不是「我動了這份文件的主題」，而是「我建立了某份文件所列舉種類的新實例」。這個觸發發生在程式碼的建立時刻，離任何文件都很遠，工作流程裡也沒有任何東西指回去。dashboard 模組的案例還有一層：該模組是*刻意、論證充分的*設計修正（依 architecture.md §7.7 的組合層，解掉一個模組循環）— 記錄在該變更自己的 design.md 裡，卻沒有傳播到列舉模組集合的那兩份常設文件，於是變更一交付，repo 的規則與 repo 的程式碼就不一致了。

**如何避免：** 當變更建立了某個可列舉種類的新實例 — 模組、Mongo collection、事件類型、錯誤代碼、環境變數 — 就 grep 文件找出該種類既有成員被列在哪裡，並在同一變更中擴充清單。截至撰寫本文，本 repo 的註冊表有：`architecture.md` §14.1（模組，由 `module-boundaries.md` 鏡像）、`ai/context/mongodb-collections.md`（collections）、`api.md` §6（錯誤代碼）、`event-schema.md` §4（事件類型）、`docker-compose.md` §5（環境變數）。只記在變更自身 design.md 裡的刻意設計偏離是記錄了一半：常設規則文件才是*下一個*工作階段最先讀的東西。

## 模式 4：衍生數字被以超過其輸入所能支撐的權威性呈現

**Bug 本身（發現 9，外加幾個佳作）：** bootstrap 近似 — 沒有轉移歷史的機台，其整個 24 小時視窗被歸給目前狀態 — 產出的回應與完整推導的資料逐位元組相同。一台只跑了 40 分鐘的機台可以回報「Operating 24h」，沒有任何東西標示它是估計值。同一模式的較小版本：`formatDuration` 把精確的 0 渲染成「<1m」（暗示一個小的非零量），無法解析的時間戳渲染成「NaNd ago」而不是誠實的「—」。

**為什麼會發生：** 這個近似本身是*有文件、刻意的*決定（設計 D2，在 api.md 行文中有註記）— 錯誤不在做這個決定，而在把它的存在侷限於人類閱讀的行文，機器閱讀的 payload 卻不帶任何痕跡。資料品質被當成文件問題處理，而不是資料問題。隱含的假設是：API 的消費者會讀過 §4.12 的但書。而同一變更中建置的儀表板就沒讀 — 因為回應裡沒有任何可以反應的東西。

**如何避免：** 當一個計算有降級模式（近似、fallback、部分覆蓋、過期快取）時，*回應*本身就該說出來 — 一個布林值或一個覆蓋範圍時間戳只花一個欄位，卻讓每個消費者都能誠實渲染（本次修復用 `≈`）。檢驗標準：如果 UI 想要在視覺上區分估計值與真實值，它做得到嗎 — 不用解析文件的前提下？做不到的話，降級模式就是無聲的，而無聲的降級模式終究會被當成事實來讀。
