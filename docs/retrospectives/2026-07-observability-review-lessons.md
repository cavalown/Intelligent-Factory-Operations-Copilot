# Observability Review Lessons (2026-07)

Covers the `/code-review` pass on the `add-observability` change (OpenTelemetry instrumentation, `nestjs-pino` structured logging, the `ifoc.events.processed` metric, the `lgtm` compose service). The review confirmed 7 findings; this document records the 5 root-cause patterns behind them, because two of them are *recurrences* of patterns this project had already written down — the more useful lesson here isn't the individual bugs, it's why documenting a pattern once wasn't enough to stop it happening again in a different file. Fixes shipped in the same change (`tasks.md` §6).

---

## Pattern 1: New code was inserted into an existing error-classification pipeline without checking it against that pipeline's own policy — a direct recurrence of a previously-documented pattern

**The bug:** `KafkaConsumerBase.onModuleInit`'s `eachMessage` handler calls `await this.handleMessage(payload)` inside a `try` block, then (as this change added) `this.recordProcessed(payload)` in the same block. `recordProcessed` re-parsed the Kafka message with `JSON.parse` and, without checking the result was non-null, immediately did `envelope.correlationId`. A message whose value is the literal 4 bytes `null` is valid JSON — `JSON.parse('null') === null` — so the parse succeeded and the very next line threw `TypeError: Cannot read properties of null`. That `TypeError` propagated to the outer `catch`, where `isDataError()` (`error-classification.util.ts`) — written to recognize exactly three types, `SyntaxError`, `mongoose.Error.ValidationError`, `mongoose.Error.CastError` — didn't recognize it, so the error was rethrown into kafkajs's retry path instead of being swallowed as an unprocessable message. A deterministic bad envelope would have retried identically forever.

**Why this happened:** This is, almost line-for-line, `docs/retrospectives/2026-07-dashboard-metrics-review-lessons.md`'s Pattern 1: "a secondary write [operation] was placed on the primary [path's] critical path, and its failure modes were never played against the pipeline's error-classification policy." That retrospective was about a *write* (a status-transition insert) landing inside the same try block as the primary projection save; this one is about a telemetry *read/record* operation landing inside the same try block as message handling. Different operation, same shape of mistake: a new piece of code was added to an existing try/catch whose catch branch has a specific, enumerated contract (`isDataError`'s three recognized types), and the new code's own failure modes were never enumerated against that contract. The question that Pattern 1 says to ask — "for each class of error my new code can throw, what does the surrounding infrastructure do with it?" — was written down in this exact repository, in a doc about this exact file's sibling concern, and wasn't asked again when a different kind of code was added to the same try block.

**How to avoid it:** The lesson from the first occurrence needs to generalize past its original shape ("don't add a *write*" to "don't add *any code that can throw*") to actually transfer. Concretely: before adding *any* statement to an existing try block that has an error-classification catch, ask what exception types the new statement can produce and check each one against the catch's enumerated list — not just "will this normally succeed," but "what does this throw on a malformed/edge-case input, and is that type in the list." If it isn't, either the new code needs its own containment (see Pattern 2's fix) or the classifier's list needs to grow, deliberately, not by accident.

## Pattern 2: A metric's "success" label was defined by "the surrounding code didn't throw," not by the domain's own definition of success

**The bug:** The `ifoc.events.processed` counter (and the `ifoc.correlation_id`/`event_id`/`event_type` span attributes) were recorded whenever `handleMessage()` resolved without throwing. But throwing was never the only way a message could be a no-op: `MachineProjectionConsumerService` returns early (no throw) for an unrecognized `eventType`, with a comment reading "skip entirely, don't mark as processed"; both `EventConsumerService` and `AlertConsumerService` catch a Mongo duplicate-key error on redelivery and return as an idempotent no-op; `MachineProjectionConsumerService` also returns early for an unknown `machineId` or an already-applied `eventId`; `AlertConsumerService` returns early when the event doesn't warrant an alert at all. Every one of these is a deliberate, correct, non-throwing skip — and every one of them was still counted as "successfully processed," because the counter's trigger condition ("didn't throw") and the domain's actual success condition ("caused a real effect") aren't the same predicate, and nothing enforced that they should be.

**Why this happened:** `Promise<void>` doesn't carry a signal. `handleMessage`'s abstract signature returned nothing, so the only information available to the caller was "did it throw" — a boolean already borrowed for an entirely different purpose (retry classification, Pattern 1 above). Reusing "didn't throw" as a proxy for "succeeded" is an easy, invisible substitution to make, because in the *common* case (an event that genuinely gets processed) the two conditions agree — the bug only shows up on the skip paths, which is exactly the set of paths a happy-path verification (this change's own task 5.1, which exercised exactly one successfully-applied event) won't exercise. One of the five miscounted skip paths — the unrecognized-`eventType` one — is itself the resurrection of a rule from an *already-archived* change (`archive/2026-07-08-kafka-consumer-reliability-hardening`'s "unrecognized event types do not... mark themselves processed"), reintroduced one layer up, in the metrics/tracing code, by someone who wrote the domain-layer version of that exact rule in the very same file a few lines above.

**How to avoid it:** When a metric or span is meant to represent a domain concept ("processed," "succeeded," "applied") rather than a control-flow fact ("didn't throw"), give the emitting code an explicit signal for that concept — a return value, a result type, an event — rather than inferring it from whether an exception occurred. Here: `handleMessage` now returns `Promise<boolean>`, `true` only on a genuine effect, and the metric only fires on `true`. More generally: before wiring a new observability signal to an existing function's control flow, ask whether that function already distinguishes "no-op" from "did work" anywhere in its own logic (it usually does — every one of the five skip paths already had its own comment explaining *why* it was a no-op) — if so, that distinction is the thing to expose, not to re-derive from the outside.

## Pattern 3: Hand-rolled plumbing reappeared in the exact place the change's own design doc said to avoid it

**The bug:** `instrumentation.ts` built each OTLP exporter's URL with a local `otlpUrl` helper that read `env.otelExporterOtlpEndpoint` and string-concatenated a hardcoded resource path (`/v1/traces`, `/v1/metrics`, `/v1/logs`), then passed that as an explicit `url` to `OTLPTraceExporter`/`OTLPMetricExporter`/`OTLPLogExporter`. Reading the actual `@opentelemetry/otlp-exporter-base` source showed every one of those exporters already does exactly this resolution itself — including checking a *signal-specific* env var (`OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` etc.) before falling back to the generic one — but only when the constructor isn't given an explicit `url`. Passing one, as the hand-rolled helper did, silently disabled that resolution for anyone who ever set a signal-specific endpoint.

**Why this happened:** This change's own design doc (`design.md` D1) states the guiding principle in so many words: "auto-instrumentation over manual spans... auto-instrumentation covers HTTP/Mongoose/kafkajs including context propagation through Kafka headers — hand-rolling that propagation is exactly the kind of infrastructure duplication [the first retrospective's] Pattern 3 warns about." That principle was applied correctly at the scale it was stated at (no manual spans, no hand-rolled Kafka header propagation, no hand-rolled pino mixin — instrumentation-pino handles trace injection and log-forwarding automatically, a design choice this same session made explicitly). But URL construction — thirteen characters of `.replace(/\/$/, '')` and a template string — didn't *look* like "infrastructure," so it didn't trigger the same "check whether the library already does this" reflex the bigger decisions got. It is the same category of mistake as Pattern 3 of the first retrospective (building something the already-in-use library provides), just at a scale small enough to not read as a design decision worth writing down — which is exactly why it slipped past a principle that was, in this very change, being actively and correctly applied to bigger decisions.

**How to avoid it:** "Does the library already do this" is a question with no size threshold — it applies as much to a thirteen-character string helper as to a retry loop. A useful trigger: any time new code reads an env var *and* a third-party client/exporter/SDK object is being constructed nearby, check whether the constructor already reads that env var itself before wiring it through by hand — reading env vars is exactly the kind of thing well-designed SDKs (especially spec-driven ones like OpenTelemetry) do for you, and a hand-written bridge is redundant by default, not by exception.

## Pattern 4: A dependency edge was added for its ordering convenience without checking its cost on the one path that actually needed to survive

**The bug:** `docker-compose.yml`'s `backend` service gained `lgtm` in its `depends_on` list. Compose's `depends_on` (with no health condition) only waits for the dependency to *start*, not become ready — which is enough for "ordering," but it still means the dependent service is blocked if the dependency fails to start at all. `grafana/otel-lgtm` is a large, multi-component image; on a fresh clone, offline, or under Docker Hub rate limiting, its pull can fail — and empirically (reproduced with a minimal repro compose file), that failure prevents `backend` from starting at all, even though nothing about `backend`'s actual runtime behavior needs `lgtm` present (every fail-soft claim in the same change's own docs and design doc is about `lgtm` being *stopped after the stack is already running*, never about it failing to start in the first place).

**Why this happened:** Startup ordering felt like free documentation value — "kafka, mongodb, and lgtm start first, then backend" is a clean sentence — and `depends_on` was the tool at hand to express it. But the change's own design goal (D5: "the backend behaves identically with or without the observability container") already implied `backend` has *zero* runtime requirement on `lgtm`, at any point, including startup — so there was nothing for the ordering to actually protect. The dependency edge was added because it was easy to add and seemed harmless, not because anything downstream needed it; its one real effect (coupling `backend`'s own startup success to an unrelated container's image pull) was never checked against the design goal that made it unnecessary in the first place.

**How to avoid it:** Before adding a `depends_on` (or any ordering/coupling primitive), ask what specifically breaks *without* it — not "would ordering be nice," but "what observable behavior requires B to exist before A." If the honest answer is "nothing, A doesn't touch B until much later and tolerates B being absent indefinitely" (which was already true here, and already written down in the same change's own design doc), the edge has a real cost (A's startup now depends on B's own success) and no offsetting benefit, and shouldn't be added.

## Pattern 5: A task was marked complete based on how much surrounding work felt done, not on whether that task's own stated scenario was actually verified

**The bug:** `tasks.md`'s task 2.3 ("confirm a forced 5xx appears in both access log and an errored trace span") was checked off `[x]` — but its own note admitted the forced-5xx repro was attempted, failed (Mongoose's 30s `serverSelectionTimeoutMS` buffering got in the way), and was never actually completed; the checkbox was flipped anyway because the surrounding work (access logging, trace correlation) genuinely was done and verified. Separately, `tasks.zh-TW.md` — this project's established convention for every `tasks.md` in `openspec/` — was never updated at all, leaving it showing all 13 tasks as not-started while the English file showed the whole change complete with detailed verification notes.

**Why this happened:** Both are instances of the same slip: treating a checklist item as a proxy for "this general area is done" rather than as a literal claim about the specific sentence next to the checkbox. Task 2.3 bundles two claims (access logging works; 5xx marks a span errored) into one checkbox — the first was thoroughly verified, and that verification's confidence bled into checking off the second, unverified claim too. The zh-TW omission is the same failure at the file level: eleven other files got bilingual updates in the same change (docs, architecture, the compose file's own doc), so the *change* felt bilingual-complete, and the one remaining file — a checklist, not prose anyone was actively re-reading — didn't get individually checked against the convention.

**How to avoid it:** A checkbox's claim is exactly the sentence next to it, not the surrounding paragraph's vibe. Before checking one off, re-read its literal wording and ask whether *that specific sentence* was verified — not whether the feature area it belongs to feels finished. For multi-language repos with a stated bilingual-sync convention, treat the sync itself as a checklist item with its own explicit verification step (e.g., a final `git status` sweep across every `.md` touched, checked against every `.zh-TW.md` twin) rather than something that's assumed to have happened because most files got it.
