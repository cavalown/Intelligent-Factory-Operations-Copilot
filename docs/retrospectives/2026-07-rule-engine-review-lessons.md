# Rule Engine Review Lessons (2026-07)

Covers the `/code-review` pass on the `add-rule-engine` change (the enrichment consumer that computes `temperatureExceedsThreshold`/`isSensorFailure` once and republishes them, replacing the duplicated per-service classification `docs/design/machine-schema.md` §5.4 had documented as an accepted MVP risk). The review confirmed/judged-plausible 10 findings across correctness, test coverage, type safety, documentation, and convention-following; all 10 were fixed in the same change (`tasks.md` §6). This document records the 6 root-cause patterns behind them. The headline pattern isn't a coding mistake at all: three of the ten findings are cases where a convention this same agent, in this same session, had already applied correctly to the *previous* change — wasn't reapplied to this change's own new artifacts. Doing something right once, even minutes earlier, didn't make it automatic the second time.

---

## Pattern 1: Consolidating logic into a new single source of truth carried the comparison, but dropped the guards that used to sit next to it

**The bug:** Two findings, same shape. First: `RuleEngineConsumerService` computed `temperatureExceedsThreshold = event.payload.temperature > machine.temperatureThreshold` with no `Number.isFinite(temperature)` check — but both call sites this logic was consolidated *from* (`MachineProjectionConsumerService`, `AlertConsumerService`) had that exact guard, sitting immediately above their own copy of the same comparison. A non-finite temperature would have been silently classified `temperatureExceedsThreshold: false` — a fabricated "within threshold" claim — rather than treated as unclassifiable the way an unknown machine already was. Second: `handleMessage` returned `true` unconditionally, including for the unknown-machine no-op, even though `ai/rules/observability-conventions.md` — a rule this same consumer's base class (`KafkaConsumerBase`) exists specifically to make easy to follow — literally enumerates "unknown machine" as a case requiring `false`.

**Why this happened:** When logic moves from N duplicated call sites into one new call site, the *positive* logic (the comparison itself) is the part that visibly needs to be written, so it gets written and gets attention. The *defensive* logic sitting next to it at the old sites — the guard clauses, the "return false for this input" branches — isn't a separate line item on any task list; it's implicit context that traveled with the comparison at the old sites and had to be actively noticed and re-added at the new one. Nothing in `tasks.md`'s original task 2.2 said "port the Number.isFinite guard too" — it only said what to compute, not what to guard against, because the guard's necessity wasn't visible from the new consumer's own code, only from the two old ones being replaced.

**How to avoid it:** When consolidating duplicated logic from N places into one, don't just diff the *positive* logic across the N sites (do they compute the same thing) — diff the *guards* around it too (do they validate the same preconditions before computing). Read the full surrounding function at each old call site being retired, not just the line being moved, and ask "what made this line safe to run *there*" before assuming the same line is safe to run in the new location without its neighbors.

## Pattern 2: A design doc's own explicit, self-imposed requirement was written in prose but never became a checklist item

**The bug:** `design.md`'s Risks section stated, discussing the deleted contract test: "Its replacement is direct test coverage on Rule Engine's own classification logic (tasks.md should include this explicitly, not leave it implicit)." This is about as unambiguous as a self-imposed requirement gets — it even names the exact failure mode to avoid ("not leave it implicit"). `tasks.md`'s Verification section (5.1–5.5) shipped anyway with only manual/live demo steps; no unit test task for Rule Engine's classification logic existed anywhere in the checklist.

**Why this happened:** This is a narrower, more literal case than the observability retrospective's Pattern 5 ("a task was marked complete based on surrounding work feeling done") — there, a checkbox existed and got checked prematurely. Here, no checkbox existed at all. The design doc was written, its Risks section flagged the requirement in the same breath as writing it, and then the very next artifact in the sequence (tasks.md) was drafted without cross-referencing that specific sentence back into a task line. The requirement was correct and explicit at the moment it was written, but "write it down as a requirement" and "convert it into a tracked task" are two different actions, and only the first one happened.

**How to avoid it:** Any sentence in a design doc's Risks/Trade-offs section that prescribes a concrete follow-up action (a test, a doc update, a migration step) is not satisfied by existing in the design doc — it needs a literal, matching line in `tasks.md` before the artifact sequence is considered complete. When drafting `tasks.md`, grep the design doc for imperative/prescriptive language ("should include," "needs," "must") and check each one has a corresponding task, rather than trusting that writing the requirement once was enough for it to happen.

## Pattern 3: A structural protection already used elsewhere in the same file for an analogous case wasn't extended to new fields of the same shape

**The bug:** `payload: TPayload` in `MachineEventEnvelope<TEventType, TPayload>` is scoped per discriminated-union member — `TemperatureReportedEvent`'s `payload` is `TemperatureReportedPayload`, `StatusChangedEvent`'s is `StatusChangedPayload`, and TypeScript's discriminant narrowing enforces this at every read site. The two new classification fields (`temperatureExceedsThreshold`, `isSensorFailure`) were added directly to the shared `MachineEventEnvelope` base instead, so every member of the `MachineEvent` union nominally carried both fields regardless of relevance — the type system couldn't have caught a future `event.isSensorFailure` read inside an `ERROR_OCCURRED` branch, even though the exact same mistake with `payload` *would* have been a compile error.

**Why this happened:** The working example of "how to scope a field to specific event types" was sitting in the same file, a few lines away, doing exactly the job the new fields needed — but it wasn't recognized as a template to copy, because it didn't look like the same problem. `payload`'s per-member scoping exists because each event type's payload shape is obviously different (different field names entirely); the new fields are a single, uniform `boolean` shape across the two members that use them, which reads as "just add it to the base, it's the same type everywhere" — the fact that *relevance*, not *shape*, is what needed scoping wasn't obvious from the field's own type signature.

**How to avoid it:** When adding a field to a discriminated union that's only meaningful for a subset of members, the question isn't "does this field have the same type everywhere" — it's "should code be allowed to read or write this field when the discriminant doesn't match." If the answer is no, scope it to the specific member types (via intersection or a member-specific interface), even if the field's shape is identical across every member it appears on.

## Pattern 4: Conventions this same agent had already correctly applied earlier in the same session weren't reapplied to this session's own later work

**The bug:** Three independent findings, same root cause. (a) None of `add-rule-engine`'s six new OpenSpec artifacts had `.zh-TW.md` twins, violating `ai/rules/bilingual-docs.md`'s explicit scope — despite this exact agent, earlier in this exact session, correctly creating full `.zh-TW.md` twins for every artifact of the *previous* change (`add-observability`) before archiving it. (b) `backend/.env.example` was never updated with the new `KAFKA_TOPIC_MACHINE_EVENTS_ENRICHED` variable, even though `docker-compose.yml`, `docker-compose.md` §5, and `local-development.md` all got it in the same registry-sweep task — `.env.example` is the same *category* of "list of Kafka topic env vars" as the other three, just a fourth location the sweep's mental checklist didn't include. (c) The new consumer group was named `rule-engine-group`, breaking the `<module>-service-group` pattern all three prior consumers followed exactly (`event-service-group`, `machine-service-group`, `alert-service-group`) — the name was derived from the feature's name ("Rule Engine"), not from the module's directory name (`rules/`), which is what the other three groups were actually keyed to.

**Why this happened:** All three are instances of applying a convention correctly when it's the explicit subject of attention, and failing to apply it when it's incidental to a different task. The bilingual-docs rule was followed meticulously during the observability change's *archival* step — a task whose entire purpose was "finish this change properly" — but this change's artifacts were created during active *drafting*, where the immediate attention was on getting the technical content right, and the bilingual requirement wasn't re-checked as a separate, later step. The `.env.example` gap happened because the registry-sweep task (4.2) was scoped by *memory of which files usually need updating*, not by a systematic search for every file matching the pattern being updated — three hits felt like a complete sweep. The naming gap happened because naming a new thing pulls from the vocabulary already in your head (the feature you're building, "Rule Engine") rather than from a checklist of "what does the convention doc say the pattern is" — especially when, unlike the three existing consumers, the module's directory name and the feature's product name aren't identical (`rules/` vs. "Rule Engine").

**How to avoid it:** Correctly applying a convention once in a session is not evidence it will be applied again for the next artifact — each new artifact needs its own explicit check against the convention, not inherited confidence from having gotten it right nearby. Concretely: (1) bilingual docs — treat "new openspec change: create zh-TW twins" as a task that happens at *artifact creation* time, not archival time; (2) registry sweeps — search for the pattern (`grep` for the sibling env var across the whole repo) instead of recalling which files "usually" need it; (3) naming — before naming anything a convention doc governs, re-read the convention doc's literal pattern and apply it to the module/directory identity, not the feature's product name, especially when the two diverge.

## Pattern 5: A canonical doc's prediction wasn't marked fulfilled once the change that fulfilled it shipped

**The bug:** `docs/design/machine-schema.md` §5.4 documented the classification-duplication problem and named its fix directly: "Phase 2's Rule Engine should do... [this]." `add-rule-engine`'s own `design.md` and several source-code comments explicitly cite §5.4 as their motivating context. The change shipped — the Rule Engine now exists, the contract test §5.4 named as the interim mitigation is deleted — but §5.4 itself was never revisited, so it kept describing an open problem and a safety net that, by the time of the code-review pass, no longer existed.

**Why this happened:** `machine-schema.md` was read *from* repeatedly while drafting this change (it's cited in the proposal, the design doc, and the Rule Engine's own source comments) but never *back to* — every reference treated it as a source of context to pull from, not a document with its own claims that this change's completion would falsify. The doc's own status wasn't tracked as a stakeholder in the change the way `tasks.md` or the specs were.

**How to avoid it:** When a design doc or proposal explicitly names an existing doc as "the problem this change solves," that existing doc is not just background reading — it has a stake in the change's completion and needs its own update once the change ships, the same way a spec's requirements get updated. Before closing out a change, grep for every doc your own proposal/design cited as "the problem" or "the reason," and check whether their claims are still accurate now that the change exists.

## Pattern 6: A design decision's benefit was described with more rigor than its cost

**The bug:** `design.md`'s Decisions section explained clearly *why* computing classification once in Rule Engine is better than two independent derivations. What it didn't quantify: for a `TEMPERATURE_REPORTED` event, the total MongoDB read count against a machine document is unchanged at 2 before and after this change (Rule Engine's new `findRaw` lookup replaces, rather than eliminates, Alert Service's old one; Machine Service's own `findOne` was never touched) — and Machine/Alert Service's processing, previously parallel across two independent consumer groups reading the raw topic directly, is now serialized behind Rule Engine's own consume-classify-produce step. Neither claim is false or hidden — both are directly inferable from reading the diff — but neither was stated in the design doc's own Risks section until code review asked the question directly.

**Why this happened:** The benefit ("one place computes the classification instead of two") is a qualitative, structural claim that's easy to state confidently without measurement. The cost (I/O count, parallelism) is a quantitative claim that requires actually counting operations across the old and new call graphs — a different, more effortful kind of analysis that a Risks section's existing bullet points (latency, contract-test retirement, future extensibility) didn't happen to prompt.

**How to avoid it:** When a design doc's Decisions section makes a structural improvement claim ("compute once instead of twice," "one source of truth instead of two"), the Risks section should include the matching quantitative check: count the actual operations (DB reads, network hops, lock acquisitions) on the critical path before and after, not just the count of *places the logic lives*. A design can still be the right call with an unfavorable count on one axis (as this one was, at this project's demo scale) — the point isn't to block the decision, it's to make sure the trade-off was actually seen before it was accepted.
