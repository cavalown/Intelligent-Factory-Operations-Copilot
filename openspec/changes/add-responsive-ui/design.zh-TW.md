# 設計：add-responsive-ui

## 脈絡

目前的前端：五個頁面、Naive UI、以桌面為前提。低於約 1024px 的已知破版：`.dashboard-row`/`.detail-row` 是不換行的 `flex 3:2`；表格（6 欄 + JSON payload 欄）溢出；頂部 `NMenu` 在約 500px 以下換行得很糟；`NDescriptions :column="3"` 太擠。統計磚已經使用 NGrid 響應式 span 且在所有寬度都存活 — 證明機制可行，只是從未套用到其他地方。2026-07-12 探索階段鎖定的決策：手機用底部分頁、混合式表格策略、Phase 1.2、Naive UI 斷點、常設規則 `ai/rules/frontend-responsive.md`。

## 目標 / 非目標

**目標：**

- 五個頁面在 390px（手機）與 768px（平板直向）都可用；桌面不變。
- 手機用底部分頁列；手機的機台列表用卡片；平板斷點以下列改堆疊；查詢型表格精簡+可捲動。
- 手機/平板路徑上的目標尺寸足以觸控。
- 三視口的 Playwright 驗證附截圖。

**非目標：**

- PWA/離線/可安裝；滑動手勢；深色模式；前端遙測；任何後端變更；重新設計桌面版。

## 決策

### D1：斷點的單一真實來源 — 基於 `matchMedia` 的 `useViewport` composable

一個單一的 composable（`isPhone` < 640、`isTablet` 640–1024、`isDesktop` ≥ 1024），建立在 `window.matchMedia` 上；元件使用具名布林值，永不使用原始寬度。**套用時修正：** 原始計畫寫「包裝 Naive UI 的斷點 hook」，但那個 hook 位於 naive-ui 的傳遞性依賴 `vooks`，不是公開 API 面 — matchMedia 守住零新依賴的承諾，又不耦合到內部實作。區間只用 min-width 查詢，因此在構造上就互補（區間邊緣沒有小數寬度縫隙）、狀態在首次使用時惰性初始化（import 永不碰 `window` — 對未來的測試環境安全），且 HMR dispose 在開發模式移除監聽器。理由：常設規則讓響應式檢查成為永久要求 — 散落的 `window.innerWidth` 或重複的斷點字面值會立刻腐化。純 CSS 的考量（堆疊、間距）用同樣 640/1024 值的普通 media query；結構性考量（渲染哪個元件）用 composable。

### D2：導覽 — 條件渲染，不是 CSS 隱藏

手機：固定底部分頁列（4 項、圖示 + 標籤、56px 高、內容獲得底部 padding）；平板/桌面：既有的頂部 `NMenu`。透過 `useViewport` 條件渲染，而不是 `display:none` 的重複 — 兩個導覽都在 DOM 會讓路由作用中邏輯加倍並混淆焦點順序。分頁列是一個小的客製元件（Naive UI 沒有 bottom-nav 原語；約 40 行，仍然零新依賴）。

### D3：機台列表 — 同一資料來源的兩種渲染

手機：卡片清單（狀態標籤 + 名稱醒目，溫度/健康/更新為緊湊列，整卡可點）。平板/桌面：既有的 `NDataTable`。同一個查詢、同一套篩選邏輯（`?status=` 下鑽完全相同 — 卡片也會篩選）。否決：CSS 變形的表格（脆弱）與響應式欄位隱藏（手機上的機台列表是*觸控*頁面；表格列是糟糕的點擊目標）。

### D4：查詢型表格 — 先精簡，再捲動

`EventsTable` 增加 `compact` 模式（由 `useViewport` 驅動）：手機上移除 payload 欄（payload 透過詳情頁一點即達；手機上截斷的 JSON 對誰都沒資訊量）、保留時間/機台/類型，並包在 `overflow-x: auto` 裡承接仍然太寬的情況。Event Center 與兩個內嵌的事件列表都經由共用元件繼承 — 一份實作、服務三個頁面（add-frontend-mvp 的重用決策開花結果）。

### D5：堆疊與描述區

`.dashboard-row`/`.detail-row` 在 1024px 以下變成 `flex-wrap`、子項全寬（只用 media query — 不用 JS）。`NDescriptions` 透過 composable 使用自己的響應式欄數（手機 1、平板 2、桌面 3）。磚格線已經行為正常；其 span 在驗證中於 390px 重新檢查，不重新設計。

### D6：驗證是三視口、以截圖為證

Playwright 流程在 390/768/1280 跑五個頁面，斷言每頁的哨兵元素並擷取截圖；手機的執行額外演練分頁列導覽與卡片點擊。桌面截圖以目視與現行行為比對（不用像素比對工具 — demo 專案）。

## 風險 / 取捨

- [機台列表的兩種渲染可能漂移（表格加了欄、卡片忘了）] → 兩者共用同一個列塑形 helper/型別；常設規則的「每個變更說明手機行為」是流程防線。
- [底部分頁列遮住內容] → 隨分頁列套用的版面 class 保留底部 padding；以截圖驗證。
- [Naive UI 元件的 padding 為桌面調校] → 可用就接受預設值；只在驗證顯示真實問題時調整（避免千刀萬剮式覆寫的主題分叉）。
- [輪詢 + 窄裝置 = 電池] → 不在範圍內；5 秒輪詢在分頁隱藏時已經暫停（vue-query 預設）。
