## 6. Code-review fixes (2026-07-19 review, high effort, 8-angle finder + verify pass, 7 findings all addressed)

- [x] 6.1 `recordProcessed` could null-deref: a Kafka message whose value was the literal bytes `null` parsed successfully (`JSON.parse('null') === null`) and the subsequent `envelope.correlationId` access threw a `TypeError` that `isDataError` doesn't recognize, reaching kafkajs's retry path for a deterministically-bad message instead of being swallowed. Fixed at the root: envelope parsing moved into `KafkaConsumerBase.onModuleInit` itself (once, before calling `handleMessage`), with an explicit `parsed === null || typeof parsed !== 'object'` check that throws `SyntaxError` (correctly classified as a data error) instead of letting a bad cast propagate a `TypeError` later.
- [x] 6.2 `ifoc.events.processed` counted "handleMessage didn't throw" as "processed," so 5 distinct non-throwing skip paths across the three consumers (unrecognized eventType, two idempotent duplicate-key no-ops, unknown machineId, stale eventId, no-alert-warranted) were miscounted as successes â€” one of them silently re-violating an already-archived spec requirement ("don't mark as processed") at a new layer. Fixed: `handleMessage`'s abstract signature changed from `Promise<void>` to `Promise<boolean>`; each subclass now returns `true` only on genuine effect and `false` on every skip path; `KafkaConsumerBase` only calls `recordProcessed` when `true`.
- [x] 6.3 `recordProcessed`'s own `JSON.parse` re-parsed the same Kafka message buffer each subclass's `handleMessage` already parsed, and its local `EventEnvelopeIdentity` interface duplicated a subset of the canonical `MachineEventEnvelope` type. Fixed by the same 6.1/6.2 refactor: the base class parses once and passes the typed `MachineEvent` to `handleMessage`, which no longer parses at all; `EventEnvelopeIdentity` deleted.
- [x] 6.4 `backend`'s `depends_on` included `lgtm`, so a failed/slow `otel-lgtm` image pull on a fresh `docker compose up` would block `backend` from starting at all â€” contradicting the "optional/fail-soft" framing for first-time bring-up specifically (empirically reproduced: `depends_on` without a health condition still requires the dependency to *start* before the dependent starts). Fixed: `lgtm` removed from `backend`'s `depends_on` entirely â€” it was never required for runtime correctness (OTLP export is fail-soft regardless of start order), only ever useful as ordering, and the ordering wasn't worth the coupling.
- [x] 6.5 `instrumentation.ts`'s hand-rolled `otlpUrl` helper always passed an explicit `url` to all three OTLP exporters, which (verified against `@opentelemetry/otlp-exporter-base` source) permanently disables the SDK's own signal-specific endpoint env var resolution (`OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` etc. would be silently ignored). Fixed: `url` no longer passed to any exporter; each resolves its own endpoint from env vars per the OTel spec's own fallback chain. `env.config.ts`'s now-unused `otelExporterOtlpEndpoint` field removed accordingly.
