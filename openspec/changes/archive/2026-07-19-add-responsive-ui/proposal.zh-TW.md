# 提案：add-responsive-ui

## 為什麼

MVP 前端當初有意識地只做桌面版（add-frontend-mvp 把響應式版面列為 Non-Goal），但工廠監控的真實使用者拿的是平板（在現場走動的操作員）與手機（瞄一眼狀態的主管/值班人員）— 在那些寬度下，今天的並排版面塌成不可用的窄欄、資料表格溢出、頂部選單壞掉。2026-07-12 決定（探索階段）：這是路線圖的 **Phase 1.2**，並搭配新的常設規則 `ai/rules/frontend-responsive.md`，讓所有未來的前端工作都必須考慮響應式。

## 改什麼

- **導覽**：手機使用底部分頁列（四個目的地、單手可及）；桌面的頂部水平選單維持不變。
- **機台列表**：手機上渲染為觸控友善的卡片（機台名稱 + 狀態標籤醒目，指標在下）— 它是主要的走動頁面；桌面表格不變。
- **查詢型表格**（Event Center、Dashboard 近期事件小工具、機台詳情事件列表）：窄寬度下精簡欄位 + 水平捲動 — 查詢頁能容忍捲動；做卡片回報低。
- **並排列**（Dashboard 的事件+告警/摘要、機台詳情的事件+摘要）：平板斷點以下垂直堆疊。
- **機台詳情的描述區**：3 欄格線在窄寬度降為 1–2 欄。
- **觸控目標**：手機/平板路徑上的互動元素以觸控尺寸設計；不使用只靠 hover 的操作提示。
- **機制**：Naive UI 的斷點系統（`useBreakpoint`、NGrid 響應式 span、640/1024）加最少量的 media query — 零新依賴。
- **不退步**：桌面（≥1024）在意圖上逐像素保留目前版面；控制室仍是一級目標。

不在範圍內：PWA/離線、原生手勢、依裝置的功能差異（全部五頁在所有寬度都存在）、前端遙測、深色模式。

## 能力

### 新能力

無。

### 修改的能力

- `operator-ui`：新增響應式行為需求（delta 增加需求；既有的頁面內容需求不變）。

## 影響

- **程式碼**：只有 `frontend/src/**` — App shell（導覽）、MachineListPage（卡片模式）、EventsTable（欄位精簡 + 捲動）、Dashboard/MachineDetail 版面 CSS、一個小的 `useIsMobile` 風格 composable。
- **無後端、API 或 compose 變更。**
- **文件**：路線圖 Phase 1.2（已隨本提案套用）；`ai/rules/frontend-responsive.md`（已建立）是本變更所實例化的常設規則。
- **驗證**：既有的 Playwright 流程在三種視口（390 / 768 / 1280），每頁截圖。
